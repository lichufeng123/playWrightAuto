# 测试用例 Review

## 目标

从测试负责人视角审查 AI 生成的测试用例，发现遗漏、错误、重复、不可执行和 PRD 不一致的问题。

## 输入

- PRD Markdown 或原文。
- 测试点 JSON。
- 测试用例 JSON。

## 输出格式

输出 Markdown 报告，包含以下部分：

```markdown
# 测试用例 Review 报告

## 总体结论

## 严重问题

## 覆盖遗漏

## PRD 不一致或疑似编造

## 不可执行/预期不可判断

## 重复或可合并用例

## 优先级调整建议

## 自动化建议

## 最终建议
```

## Review 规则

1. 先看 P0 测试点是否都有对应用例。
2. 检查用例是否能追溯到 PRD 或测试点，不能追溯的标记为“疑似编造”。
3. 检查步骤是否可执行，避免“验证功能正常”这种废话步骤。
4. 检查预期结果是否可判断，不能只写“显示正确”。
5. 检查一条用例是否验证过多目标，必要时建议拆分。
6. 检查异常、边界、权限、状态流转、数据一致性是否遗漏。
7. 检查优先级是否合理，核心链路和高风险规则应为 P0/P1。
8. 检查自动化建议是否现实：稳定、可重复、数据可控的用例才适合优先自动化。

## 输出要求

- 问题按严重程度排序。
- 每个问题给出原因和修复建议。
- 如果无法判断，明确写“需人工确认”，不要装懂。
- 不要重写全部测试用例，只输出 Review 意见。
