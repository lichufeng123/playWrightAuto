# 测试用例 Review

## 目标

从测试负责人视角审查 AI 生成的测试用例，发现遗漏、错误、重复、不可执行和 PRD 不一致的问题。

## 输入

- PRD Markdown 或原文。
- 测试点 JSON。
- 测试用例 JSON。

## 输出格式

输出 Markdown 报告，包含以下部分：

```markdown
# 测试用例 Review 报告

## 总体结论

## 严重问题

## 覆盖遗漏

## PRD 不一致或疑似编造

## 不可执行/预期不可判断

## 重复或可合并用例

## 自动化建议

## 最终建议
```

## Review 规则

1. 先看每个测试点是否都有对应用例。
2. 检查用例是否能追溯到 PRD 或测试点，不能追溯的标记为“疑似编造”。
3. 检查步骤是否可执行，避免“验证功能正常”这种废话步骤。
4. 检查预期结果是否可判断，不能只写“显示正确”。
5. 检查一条用例是否验证过多目标，必要时建议拆分。
6. 检查异常、边界、权限、状态流转、数据一致性是否遗漏。
7. 检查自动化建议是否现实：稳定、可重复、数据可控的用例才适合优先自动化。
8. 不要输出“优先级调整建议”章节，因为测试用例模板中没有优先级字段。

## 人工审核列

以下章节必须使用 Markdown 表格，并包含 `审核意见` 列：

- 严重问题
- 覆盖遗漏
- PRD 不一致或疑似编造
- 不可执行/预期不可判断
- 重复或可合并用例
- 自动化建议

表格建议字段：

```markdown
| 编号 | 关联用例/测试点 | 问题说明 | 修复建议 | 审核意见 |
|---|---|---|---|---|
| 1 | TC-001 | xxx | xxx | 待人工审核 |
```

`审核意见` 默认写 `待人工审核`，方便测试人员后续改为 `修改`、`忽略`、`已处理`、`需产品确认` 等。

## 去重规则

- 同一个核心问题不要在多个章节重复出现。
- 如果同一条用例/测试点同时存在“PRD 不一致”和“不可执行”，只保留更贴近根因的章节。
- 如果重复问题确实涉及多个章节，合并到一个表格行中，在 `问题说明` 中写清多重原因。
- `关联用例/测试点` 相同且 `问题说明` 核心含义相同的，必须合并。
- 不要让测试人员对同一个问题审核两次。

## 输出要求

- 问题按严重程度排序。
- 每个问题给出原因和修复建议。
- 如果无法判断，明确写“需人工确认”，不要装懂。
- 需要人工介入的建议必须出现在带 `审核意见` 列的表格中。
- 不要输出“优先级调整建议”章节，不要讨论 P0/P1/P2 调整。
- 不要重写全部测试用例，只输出 Review 意见。
