# PRD 测试点重构

## 目标

把 PRD 原始信息转换成原子测试点清单，不做测试用例细节，不编造 PRD 没有的信息。

测试点阶段只回答“这段需求需要拆成哪些可测点”。不要展开成测试用例，不要输出测试步骤、前提条件、预期结果、实际结果、状态。

## 输入

- PRD Markdown 或 PRD 原文。
- 原始 XMind JSON，通常是忠实还原 PRD 结构的思维导图。
- 可选：UI 截图、页面说明、已有测试经验。

## 输出要求

只输出 JSON，不输出 Markdown 代码块。结构如下：

```json
{
  "summary": "本次测试点提取摘要",
  "risk_notes": ["风险或待确认问题"],
  "test_points": [
    {
      "id": "TP-001",
      "module": "模块名称",
      "modulePath": ["一级模块", "二级标题", "小标题"],
      "sourceTitle": "PRD标题或小标题",
      "testPoint": "原子测试点",
      "source": "对应 PRD 原文或 XMind 节点摘要"
    }
  ]
}
```

## 提取规则

1. 以原始 XMind 的标题层级为准，不要打乱模块归属。
2. 每条测试点必须写 `modulePath`，例如 `["5.1.5 分享拉新", "5.1.5.1 个人中心优化分享", "分享"]`。
3. 每条测试点必须写 `sourceTitle`，通常取 `modulePath` 最后一层，例如 `分享`。
4. 把一条 PRD 长句拆成多个原子测试点，一点只表达一个可验证能力。
5. `testPoint` 要短、清楚、像清单项，例如“可分享落地页广告信息”“广告信息可同步到手机端”“点击【邀请用户】跳转分享页面”。
6. 不要照抄 PRD 长句，不要写成完整测试用例，不要带“验证是否”“检查是否”等空话。
7. 不要输出 `priority`、`type`、`feature`、`precondition`、`steps`、`expected`、`actual`、`status` 等测试用例或分类字段。
8. 如果一句需求包含多个动作、对象或结果，必须拆开。例如“跳转分享页面，支持复制邀请码/邀请链接，成功后显示提示”至少拆成“跳转分享页面”“支持复制邀请码”“支持复制邀请链接”“复制成功提示”。
9. 如果需求中出现括号补充条件，也要拆成独立测试点，例如“需手机端适配显示”拆成“手机端适配展示”。

分享步骤：点击[邀请用户]，跳转分享页面，支持在本页面复制个人邀请码/邀请链接，成功后显示“已成功复制链接”；

## 质量门槛

- 每条测试点都要能被后续转成测试用例。
- 每条测试点都要有来源，不确定来源时写“需人工确认”。
- 每条测试点都要能通过 `modulePath` 追溯到所属标题；没有模块路径的结果视为不合格。
- 一个测试点不能同时包含两个以上验证目标。
- 不要输出与 PRD 无关的行业常识扩展。
